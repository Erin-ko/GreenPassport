# 綠色循環經濟護照 (Green Cycle Passport) 測試規格與計劃書

## 1. 測試目標與範圍 (Goals & Scope)
- **測試目標**：確保系統之會員登入認證、物品數位護照管理、健康度保養提醒、微型零件社群募集看板等核心 MVP 功能運作無誤，並符合 3 天內產出 Demo 系統的性能與資安隱私需求。
- **在測試範圍內 (In Scope)**：
  - 會員註冊、登入與 JWT 認證流程。
  - 物品護照的建立（CRUD）與健康度、碳排估算。
  - 物品相片 Base64 上傳限制與壓縮邏輯。
  - 到期與保養提醒觸發。
  - 社群互助貼文發布、查詢及距離模糊化計算。
  - 前後端打包整合與靜態 Serving 託管。
- **不在測試範圍內 (Out of Scope)**：
  - 第二階段之年度 ESG 成就視覺化統計與分析。
  - 自動健康度進度條動態扣減。
  - 串接第三方圖床（Cloudinary）與主動發信提醒（LINE/Email）。
  - 大量使用者併發之壓力測試與效能負載測試。

## 2. 測試策略 (Test Strategy)
- **單元測試 (Unit Test)**：使用 Python Pytest 測試後端核心商業邏輯（如 Haversine 距離計算、碳排估算、密碼 Hash 強度、到期日驗證）。
- **整合與 API 測試**：利用 FastAPI TestClient 模擬 HTTP 請求，驗證成功與各類異常情境（如 422 欄位缺失、401 未授權、404 資源不存在）。
- **前端網頁 UI 測試**：手動或自動化驗證純 HTML 頁面在瀏覽器（含手機版）的 RWD 呈現、Vanilla JS 表單輸入驗證、彈出提醒 Modal 以及呼叫 Fetch API 的資料綁定。
- **端對端測試 (E2E) & 手動驗收測試**：開發完成後，由開發者或使用者依照典型使用情境，手動在瀏覽器（含手機版 Chrome/Safari）進行完整操作鏈驗證。

## 3. 測試環境與資料規則 (Environment & Test Data)
- **本機測試環境 (SQLite)**：
  - 每次執行自動化測試時，皆會重建一個乾淨的 `test.db` SQLite 檔案。
  - 測試結束後自動清理，確保測試資料不污染開發資料。
- **生產測試環境 (Render PostgreSQL)**：
  - 部署至 Render 後，於預備（Staging）或正式環境資料庫上執行 Alembic Migration，確保 Table Schema 完全同步。
  - 準備專屬的測試測試帳號（如 `test_user@example.com`）進行手動驗收測試。
  - 測試資料必須與真實使用者隔離，測試完成後手動或透過腳本清理。

## 5. 各模組測試重點 (Targeted Testing Focus)

### 5.1 FastAPI 後端 API 測試
- **驗證成功 (200/201)**：確保符合 REST 語意，並回傳正確的 Json Schema。
- **輸入欄位驗證失敗 (422 Unprocessable Entity)**：傳送錯誤型態，後端需攔截並回傳清楚的欄位錯誤提示。
- **未授權 (401 Unauthorized / 403 Forbidden)**：未帶 Token 或帶入過期 Token 請求 API，系統需予以攔截並拒絕存取。
- **找不到資源 (404 Not Found)**：請求不存在的 ID，必須安全回傳 404，不得洩漏後端代碼堆疊軌跡 (Stack Trace)。
- **伺服器錯誤 (500 Internal Server Error)**：確保後端不對外顯示詳細錯誤，只回傳通用錯誤訊息。

### 5.2 前端網頁與 UI 測試
- **畫面呈現與 RWD**：切換至手機視窗寬度（如 375px），選單與表單不能有破版或重疊。
- **表單驗證**：在密碼欄位填寫不足 8 字元、Email 未填 `@` 時，前端必須在送出前阻擋並顯示警告提示。
- **載入與錯誤狀態**：在 API 響應延遲時，畫面需顯示 Loading 提示；若 API 連線中斷，畫面上必須友善提示「網路異常，請稍後再試」。

### 5.3 資料庫測試
- **約束條件限制**：嘗試寫入 `user_id` 不存在的物品，或是 `email` 重複的資料，資料庫必須報出外鍵約束/唯一性錯誤。
- **資料庫遷移 (Migration)**：執行 Alembic upgrade/downgrade，確認資料表在升降版後不會損毀或導致資料結構錯亂。

### 5.4 安全性測試
- **憑證與機密不落地**：掃描代碼與 `.env`，確保無任何明文祕鑰被上傳。
- **SQL 注入防禦**：確保 SQLAlchemy ORM 已對參數進行安全綁定，防範 SQL Injection。

## 6. 缺陷回報格式與完成標準 (Bug Report & DoD)
- **缺陷回報模板**：
  - **標題**：[模組] 簡短描述 (例如: [物品護照] 上傳 200KB 圖片卻顯示超限錯誤)
  - **重現步驟**：1. 登入系統 2. 點擊新增 3. 上傳 200KB 圖檔 4. 點擊送出
  - **預期結果**：上傳成功並建立物品。
  - **實際結果**：彈出「圖片限制 500KB 以下」錯誤，且 API 回傳 422。
  - **環境資訊**：Chrome (v120) / iOS 17。
- **測試完成與發佈標準 (DoD)**：
  - [ ] 核心 MVP 測試案例（TC-001 ~ TC-009）通過率達 100%。
  - [ ] 無 Block 級或 High 級的未修復缺陷。

  - [ ] 程式碼皆符合資安規範（無 Secret 寫死）。
